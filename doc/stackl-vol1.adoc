== Preface
This manual provides a comprehensive description of the Stackl virtual
machine architecture and its programming model. It is intended to serve
as the primary reference for system programmers, compiler developers,
and students engaged in operating system and low-level programming coursework.

The Stackl environment is designed as a virtual machine to facilitate the
teaching of operating system concepts including interrupt handling, system
calls, and privileged operations without requiring physical hardware. By
simulating a stack-based CPU architecture, Stackl offers an accessible
yet realistic platform for experimentation and development.

Readers are encouraged to familiarize themselves with the conventions and
terminology defined in this manual to fully leverage the capabilities of the
Stackl environment.


=== About This Book
This book is part of a multivolume work entitled the *STACKL Architecture*.

STACKL defines a virtual stack-based machine environment designed for
educational and research use in operating system and systems programming.

This volume specifies the architectural state, execution model, memory
organization, and interrupt mechanism of the STACKL virtual machine.

Subsequent volumes provide a detailed instruction set reference and a system
programming guide covering runtime conventions, programming interfaces, and
software development considerations.

This volume is intended for system architects, hardware implementers,

=== Scope of This Volume
This volume describes the architectural state visible to software running on
STACKL, including the organization of its stacks, registers, program counter,
and condition flags.

It defines the execution model, memory address space, and interrupt vector
mechanism, establishing the foundation for software development on the
STACKL virtual machine.

Details of instruction encoding and semantics are covered in *Volume 2:
Instruction Set Reference*, while system-level programming conventions and
runtime environment are described in *Volume 3: System Programming Guide*.

== 1 Overview

=== 1.1 Introduction

==== 1.1.1 STACKL Features
STACKL is a virtual stack-based architecture that executes programs in a flat,
linear memory model. The stack is used for all operand storage and intermediate
computations, while instructions are fetched from the same linear memory space.

Major architectural features include:

* A flat linear memory model used for both code and data.
* A stack-based execution model with no general-purpose registers.
* Special-purpose registers for program sequencing, stack management, and system control.
* Support for 16 interrupt vectors, including reserved vectors for machine check and trap.
* User-mode memory protection with base and limit registers.
* Optional virtual memory support via page tables controlled by the VMEM_MODE flag.
* Optional hardware floating-point support controlled by the FPU_ENABLE flag.

==== 1.1.2 Registers
STACKL defines a set of special-purpose registers responsible for instruction
sequencing, stack behavior, memory protection, interrupt handling, and optional
virtual memory.

These include:

* **Instruction Pointer (IP)** – Address of the next instruction to execute.
* **Stack Pointer (SP)** – Address of the next unused stack location.
* **Frame Pointer (FP)** – Reference point for function call frames and local variables.
* **Flag Register (FLAG)** – Processor mode bits, interrupt status, and control flags.
* **Interrupt Vector Base (IVEC)** – Base address of the interrupt vector table.
* **Base Pointer (BP)** – Start of the user-mode address range.
* **Limit Pointer (LP)** – End of the user-mode address range.
* **Page Table Base Register (PTBR)** – Base address of page tables when virtual memory is enabled.

Register definitions and behaviors are detailed in Chapter 2.

==== 1.1.3 Instruction Set
The STACKL instruction set operates primarily on stack values. All standard
arithmetic, logical, and comparison instructions implicitly pop operands from the
stack and push their results back onto it.

Categories include:

* Stack manipulation operations
* Integer arithmetic and logic
* Control flow instructions
* Memory access operations
* Interrupt and trap control instructions
* System instructions affecting processor state and modes
* Floating-point instructions when the FPU is enabled

Encoding formats and instruction semantics are specified in Volume 2.

==== 1.1.4 Memory Model
STACKL uses a flat, byte-addressable memory space. Code, static data, dynamic
data, and the operand stack all reside within this single linear address space.

Program instructions are fetched by the IP from memory; the stack is simply a
designated region that grows toward increasing addresses.

In user mode, all memory accesses are checked against the Base Pointer (BP) and
Limit Pointer (LP) to enforce process isolation. In privileged mode, addressing is
absolute.

When virtual memory mode is enabled (VMEM_MODE), memory accesses are translated
using page tables whose base address is the PTBR.

==== 1.1.5 Floating-Point Unit (FPU)
STACKL optionally supports hardware floating-point operations.

Floating-point capabilities are enabled when the FPU_ENABLE bit in the FLAG
register is set. Floating-point instructions use the same stack-based operand
model as integer instructions.

Additional floating-point details appear in Volume 2.

=== 1.2 Notational Conventions

This section defines the conventions used throughout this manual for
representing numerical values, bit fields, pseudocode, instruction
formatting, and diagrams.

Unless otherwise noted, all numeric values are *unsigned* and all memory
addresses refer to *word*-addressed memory.

==== 1.2.1 Numeric Formats

The following prefixes indicate literal bases:

* `42` — Decimal (no prefix)
* `0x` — Hexadecimal
* `0b` — Binary
* `'c'` — ASCII character literal representing the byte value of *c*

Examples:

```
42         ; decimal
0x2A       ; hexadecimal
0b101010   ; binary
'\n'       ; ASCII newline
```

Hexadecimal is used for all device registers, bit masks, and memory-mapped
addresses unless stated otherwise.

==== 1.2.2 Bit Numbering and Ranges

Bits are numbered with *least significant bit* (LSB) = 0 and increase
leftward toward the *most significant bit* (MSB).

A range written as `X:Y` denotes bits X through Y (inclusive), with X ≥ Y.

Examples:

```
bit 0           ; least significant bit
bit 31          ; most significant bit of a 32-bit value
bits 7:0        ; low byte
bits 31:16      ; high halfword
```

Bit masks in tables use hexadecimal notation.

==== 1.2.3 Pseudocode

Execution semantics throughout this manual use a C‑like pseudocode.
Assignments modify architectural state unless noted.

General conventions:

* `Memory[A]` — load/store a *word* at address `A`.
* `Memory8[A]` — load/store a *byte* at address `A`.
* Register names (`IP`, `SP`, `FP`, `FLAG`, etc.) refer to CPU registers.
* Temporary symbols (`tmp`, `val`) do not represent architectural storage.

Example:

```
val = Memory[SP-1]
SP--
IP = dest
```

Control flow changes are represented by assigning to `IP`.

==== 1.2.4 Instruction Format Notation

Instruction definitions use the form:

```
OPCODE <param>
```

Parameters represent *immediate* values stored in the word after the opcode.
Multiword instructions follow these rules:

* `<param>` is a 32‑bit signed word at `Memory[IP+1]`.
* Execution increments `IP` by 2 words unless control flow overrides it.

Stack effects are shown using bracket notation:

```
Before: [..., a, b]   ; b at top of stack
After:  [..., result]
```

==== 1.2.5 Stack Diagrams

Stack diagrams depict the stack growing toward *higher* addresses.
`SP` always points to the *first unused word* above the top element.

Example:

```
Higher addresses →
+---------+  ← SP (first unused)
|    x    |
+---------+
|    y    |
+---------+
|    z    |
+---------+  ← bottom
Lower addresses
```

Function-call and interrupt-frame diagrams later in this volume follow
this same direction convention.

==== 1.2.6 Diagrams and Tables

===== Bit-Field Diagrams

Bit-field diagrams place the MSB on the left and the LSB on the right.
Ranges are displayed above the fields.

Example:

```
31                   16 15                    0
+----------------------+-----------------------+
|     high[31:16]      |     low[15:0]         |
+----------------------+-----------------------+
```

===== Tables

Tables use fixed-width column alignment unless specified otherwise.
Hexadecimal constants use the `0x` prefix, and bit positions refer to the
bit-numbering conventions defined in <<1.2.2 Bit Numbering and Ranges>>.

== 2 Basic Execution Environment

This chapter describes the fundamental execution environment of the
STACKL architecture. It covers the processor's execution model, memory
model, operational modes, special-purpose registers, and the basic rules
governing interrupts and exceptions. These elements form the basis for
all instruction semantics described in later chapters.

=== 2.1 Execution Model

The STACKL architecture is a stack-based machine. All arithmetic,
logical, comparison, and control-transfer operations use the stack as
their primary operand source and destination.

==== 2.1.1 Operand Stack

The processor maintains a unified operand stack that serves as the
evaluation structure for most instructions. The stack grows toward
increasing memory addresses.

* The **Stack Pointer (SP)** always points to the first unused word at
the top of the stack.
* Push operations store a word at `MEM[SP]` and increment `SP` by 1.
* Pop operations decrement `SP` by 1 and return the word at that
location.

Unless otherwise stated, instructions:

* Pop their input operands from the stack, in right-to-left order.
* Push their result(s) back onto the stack.

==== 2.1.2 Control Flow

Control flow is governed by the **Instruction Pointer (IP)**, which
contains the address of the next instruction to execute.

Execution proceeds sequentially unless modified by:

* Branch or jump instructions.
* Call and return instructions.
* Interrupts, traps, or machine checks.

No architectural pipeline, delay slots, or speculation behaviors are
assumed by this volume.

=== 2.2 Memory Model

STACKL uses a flat, byte-addressable **32-bit linear address space**.

* Code and data share the same address space.
* Instructions are fetched directly from memory.
* Words are 32-bit aligned.
* All stack operations use 32-bit words.

When running in privileged mode, all memory accesses use absolute addresses.
When running in user mode, memory accesses are relocated using the
**Base Pointer (BP)** and restricted by the **Limit Pointer (LP)**.

==== 2.2.1 Privileged Mode Addressing

In privileged mode:

* All load, store, and instruction fetches use absolute addresses.
* `BP` and `LP` are ignored.
* The entire 32-bit address space is available.

==== 2.2.2 User Mode Addressing

In user mode:

* The effective address for memory access is `BP + offset`.
* Accesses must lie within `[BP, LP - 1]`.
* Violations generate a machine check interrupt.

This mechanism enables simple per-process memory isolation.

=== 2.3 Processor Modes

The processor supports several execution modes that govern access to
memory, instructions, and privileged functionality. These modes are
encoded in the **FLAGS** register.

==== 2.3.1 Privileged Mode

Privileged mode is the privileged execution mode:

* Full access to memory.
* All instructions permitted.
* Interrupts may be enabled or disabled by software.
* Used for operating system code, interrupt handling, and trap handling.

Privileged mode is entered at reset.

==== 2.3.2 User Mode

User mode is the restricted execution mode:

* Memory access is limited to the range specified by `BP` and `LP`.
* Certain privileged instructions are not permitted.
* System-mode code is invoked only through traps or interrupts.

==== 2.3.3 Interrupt Mode

Interrupt mode is active when the processor is servicing an interrupt.

* Entered automatically on interrupt dispatch.
* Stack frames are created according to the interrupt rules.
* Re-entry is inhibited until the handler completes.
* Local variables are not permitted in interrupt routines; the handler
must preserve stack structure.

Interrupt mode is exited via the `RTI` instruction.

=== 2.4 Registers

The STACKL architecture defines eight special-purpose registers that
control execution flow, memory addressing, processor state, and
interrupt handling. No general-purpose registers exist; all computation
operates on the stack.

This section describes each register’s function, size, and operational
details.

[cols="10%,90%",options="header"]
|===
| Register | Description

| IP
| A 32-bit register containing the address of the next instruction to execute.
Execution proceeds sequentially by default. Branches, calls, and
interrupts update the IP to transfer control.

| SP
| A 32-bit register pointing to the first unused word at the top of the
stack. The stack grows toward increasing addresses. Push operations
store data at `MEM[SP]` and increment `SP`. Pop operations decrement
`SP` and read from `MEM[SP]`.

| FP
| A 32-bit register pointing to the base of the current function's stack
frame, typically the location of the first local variable.
Instructions that reference local variables use the FP as a base.

| FLAGS
| A 32-bit register holding processor status and mode bits. See
<<flags_register, Flags Register>> for bit definitions.

| IVEC
| A 32-bit register holding the absolute address of the interrupt vector
table. The table contains sixteen 32-bit interrupt vectors. The IVEC
register is set to zero on reset.

| BP
| A 32-bit register used in user mode to relocate memory accesses. Effective
addresses are computed as `BP + offset`. BP defines the start of a process’s
memory region.

| LP
| A 32-bit register defining the upper bound of the user process’s
accessible memory. Valid memory accesses must be in the range
`[BP, LP - 1]`. Accesses outside this range cause a machine check.

| PTBR
| A 32-bit register holding the root address of the page tables when
virtual memory mode (`VMEM_MODE`) is enabled. Used to translate
virtual addresses to physical addresses.

|===

==== 2.4.2 Instruction Pointer Register

The IP holds the 32-bit address of the next instruction to be executed.

Pseudocode for sequential execution:

```

while (not halted) {
	instruction = MEM[IP]
	IP := IP + 1
	execute(instruction)
}

```

Control flow instructions modify IP explicitly.

==== 2.4.3 Stack Pointer (SP)

SP points to the first free word on the stack, which grows upward.

Pushing a word `value` onto the stack:

```

MEM[SP] := value
SP := SP + 1

```

Popping a word from the stack:

```

SP := SP - 1
value := MEM[SP]

```

All stack operations are 32-bit aligned.

==== 2.4.4 Frame Pointer (FP)

FP points to the base of the current function’s stack frame.

Local variables and function parameters are accessed relative to FP.

During a function call:

1. Caller pushes arguments (right to left).
2. CALL instruction saves caller’s FP.
3. Callee sets FP to SP at function entry.

On return, FP is restored.

==== 2.4.5 Interrupt Vector Base Register (IVEC)

IVEC holds the 32-bit base address of the interrupt vector table.

Each vector is a 32-bit word containing the absolute address of the
interrupt handler.

At reset, IVEC is zero.

==== 2.4.6 Base Pointer (BP) and Limit Pointer (LP)

BP and LP define the user-mode address space bounds.

Effective user-mode address = BP + logical address

Valid addresses satisfy:

```

BP <= address < LP

```

Access outside this range triggers a machine check.

==== 2.4.7 Page Table Base Register (PTBR)

PTBR holds the root address of the page table structures used for
virtual memory translation when `VMEM_MODE` is set.

Page tables translate virtual addresses to physical addresses transparently.


=== 2.5 Exceptions and Interrupts (Overview)

The execution environment supports:

* Machine checks
* Trap-generated exceptions
* Up to 14 hardware interrupt sources (in addition to machine check and trap)

Interrupts are non-preemptive: the processor does not interrupt an
interrupt handler.

Pending interrupts are indicated in the high bits of the FLAGS register.
The highest-priority pending interrupt is always serviced first.

The interrupt vector table is located at the address stored in `IVEC`.
Each vector contains a 32-bit absolute address of the handler.

Detailed interrupt and trap semantics are covered in Chapter 4.

=== 2.6 Virtual Memory (Overview)

The STACKL architecture optionally supports paged virtual memory. When
the `VMEM_MODE` flag is set:

* Memory references use address translation.
* The page tables are rooted at the address contained in `PTBR`.
* Exceptions may occur on invalid or protected accesses.

This feature is optional and may not be present in all implementations.

Further details appear in a later chapter dedicated to virtual memory
operations.
